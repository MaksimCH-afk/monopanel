# Анализатор контентного gap по сущностям и интенту (MVP)

Сравнивает вашу страницу со страницами конкурентов по одному поисковому запросу
и показывает, **чего не хватает**: отсутствующие/слабо раскрытые сущности,
привязку к Google Knowledge Graph (mid) и совпадение интента. Поверх этого —
**фразовый слой** (n-граммы: слова, биграммы, триграммы, 4-граммы) с плотностью,
считающийся полностью в коде и работающий на любом языке (в т.ч. CJK, арабский).

Отдельный сервис (backend + web-фронт на одном порту), встраивается в дашборд
как карточка — по аналогии с сервисами на портах `:333x`.

## Архитектура

```
Форма (запрос + моя страница + N конкурентов)
   → препроцессинг текста (нормализация пробелов; чистку мусора делает оператор при вводе)
   → определение языка (franc-min) + сегментация (Intl.Segmenter / ICU)
   ├─ ТРЕК СУЩНОСТЕЙ  → Google NL analyzeEntities (или кодовый fallback по плотности)
   └─ ТРЕК ФРАЗ [КОД] → n-граммы (1..4) → плотность → стоп-фильтр
   → Агрегатор [КОД] — консенсус, дифф missing/weak, приоритеты, объёмы  ← числа авторитетны
   → OpenAI Structured Outputs [LLM] — интент + текстовые рекомендации     ← только текст
   → Отчёт: РАЗДЕЛЬНЫЕ секции entities_gap + phrase_gap + объём/плотность
```

**Два трека, один агрегатор.** Трек сущностей отвечает «какие темы» раскрывают
конкуренты (Google NL, salience), трек фраз — «какими буквальными словами и
насколько плотно» (n-граммы, плотность, только код). Оба прогоняются через один
и тот же движок консенсуса/диффа, но показываются **раздельно** — сводить их в
один рейтинг нельзя.

**Разделение ответственности:** весь счёт (сущности, фразы, coverage, salience,
плотность, объёмы, приоритеты) делает код. LLM только классифицирует интент и
пишет рекомендации к уже готовым спискам — он не может добавить сущность/фразу
вне списка (бэкенд отбраковывает лишнее) и не переопределяет числа.

## Стек

- Node.js ≥ 20, Express. Сегментация — встроенный `Intl.Segmenter` (ICU, полные
  данные ICU есть в Node ≥ 20 / нужен `full-icu` в урезанных образах). HTTP —
  встроенный `fetch`. Внешние зависимости: `express`, `franc-min` (определение
  языка), `stopwords-iso` (мультиязычные стоп-слова). Фронт — vanilla JS/CSS.

## Запуск

```bash
cp .env.example .env      # заполните ключи (или оставьте пустыми — включится mock)
npm install
npm start                 # http://localhost:3340
```

### Mock-режим (без ключей)

Если `GOOGLE_NL_API_KEY` / `OPENAI_API_KEY` пусты — соответствующий сервис
работает на детерминированной заглушке, и приложение проходит весь поток
end-to-end без внешних вызовов. Принудительно: `MOCK_MODE=true`.

```bash
MOCK_MODE=true npm start
```

### Docker

```bash
docker build -t content-gap-analyzer .
docker run -p 3340:3340 --env-file .env content-gap-analyzer
```

## Ключи и безопасность

Ключи Google NL и OpenAI хранятся **только на сервере**. Клиент к внешним API не
обращается — только к `/api/analyze` этого бэкенда. Ключ в браузер не
возвращается: статус показывается в маскированном виде (`AIz••••7890`).

Два способа задать ключ:
1. **В интерфейсе** — карточка «Ключи API» на странице: вставить ключ, нажать
   «Проверить» (живой мини-запрос к API) и «Сохранить». Ключ применяется сразу и
   сохраняется в `CONTENT_DATA_DIR/keys.json` (в Docker — том `/data/content`,
   переживает ребилд). Рантайм-ключ имеет приоритет над env.
2. **Через env** — `GOOGLE_NL_API_KEY` / `OPENAI_API_KEY` (см. `.env.example`).

Пустой ключ → соответствующий сервис в mock-режиме.

Эндпоинты ключей: `POST /api/keys` (задать), `POST /api/test-key` (проверить),
`GET /api/health` (маскированный статус).

| `PRIORITY_W_COVERAGE/W_MID/W_GAP` | 0.5/0.2/0.3 | веса скоринга приоритета (сущности) |
| `PRIORITY_HIGH/MEDIUM` | 0.66/0.4 | границы бакетов high/medium/low |
| `NGRAM_MAX` | 4 | верхний уровень n-грамм (n=1..MAX; триграммы обязательны) |
| `NGRAM_MIN_COUNT` | 2 | минимум вхождений n-граммы в документе |
| `NGRAM_TOP_N` | 100 | cap топ-фраз на уровень n на документ |
| `PHRASE_SALIENCE_MIN` | 0.0005 | нижний порог плотности фраз (аналог `SALIENCE_MIN`) |

| `PHRASE_SPECIFICITY_BONUS` | 0.05 | бонус специфичности за уровень n выше биграммы |
| `PROFILE_PRIORITY_W_COVERAGE/W_CENTRALITY/W_MID` | 0.5/0.3/0.2 | приоритет в режиме «только конкуренты» (без gap) |

## Режимы анализа

Перед запуском оператор выбирает режим (переключатель в форме):

- **A — «Сравнение с моей страницей»** (`mode: "compare"`, по умолчанию): нужна
  моя страница; на выходе дифф — `missing` / `weak` (сущности) и
  `phrase_gap.missing/weak` (фразы).
- **B — «Только конкуренты»** (`mode: "competitors_only"`): моя страница не нужна
  (если прислана — игнорируется); на выходе консенсусный профиль-бриф —
  `consensus_profile` (сущности) и `phrase_profile` (фразы) с полями
  `name`/`phrase`, `n`, `coverage`, `median_salience`/`median_density`, `mid`,
  `priority`, `recommendation`. Полей `target_*`/`gap` нет. Приоритет — без gap:
  `coverage + центральность (+mid)`. `volume` — медиана и распределение по
  конкурентам.
- **C — «Одна страница»** (`mode: "single"`): аудит одной статьи; конкуренты не
  нужны. На выходе профиль самой страницы — `consensus_profile` / `phrase_profile`
  (ранжирование по salience/плотности, без `coverage`) и `volume` этой страницы
  (`words`, `sentences`, `lexical_density`).

Консенсус, порог `K`, извлечение сущностей/фраз — общие для всех режимов;
разница только в наличии шага сравнения / числе документов.

## История анализов

Каждый анализ автоматически сохраняется на сервере
(`CONTENT_DATA_DIR/history.json`, последние `HISTORY_MAX`) и доступен в блоке
«История анализов»: открыть прошлый результат или удалить. Эндпоинты:
`GET /api/history` (список), `GET /api/history/:id` (полный результат),
`DELETE /api/history/:id`. Хранится результат (числа + рекомендации), не сырой
вставленный текст.

## API

- `GET /api/health` — статус, флаги mock, `max_competitors`.
- `POST /api/analyze` — тело по §7.1 ТЗ (+ опциональные `custom_stopwords` и
  `mode`). Ответ обратносовместим — все прежние поля трека сущностей сохранены,
  добавлены:
  - `mode` — эхо режима анализа (`compare` | `competitors_only`);
  - режим A: `phrase_gap: { missing, weak }` (элементы с `phrase`, `n`,
    `coverage`, `median_density`, `target_density`, `gap`, `priority`,
    `recommendation`); режим B: `consensus_profile` + `phrase_profile`;
  - `volume` расширен: `sentences`, `median_competitor_sentences`,
    `lexical_density`, `median_competitor_lexical_density` (в режиме B — медианы
    и распределение по конкурентам);
  - `language: { target, dominant }` — определённый язык (ISO 639-3);
  - `entities_mode` — `nl` | `code` | `mixed` (трек сущностей: NL API или кодовый
    fallback по плотности).

## Логика приоритета

```
gap        = missing → 1;  weak → clamp((median − target)/median, 0, 1)
score      = W_COVERAGE·(coverage/N) + W_MID·has_mid + W_GAP·gap
приоритет  = score ≥ HIGH → high;  ≥ MEDIUM → medium;  иначе low
```

## Отказоустойчивость

Сбой одного конкурента не роняет анализ — он исключается, счётчик
обработанных/пропущенных и предупреждение отражаются в отчёте. Сбой «моей
страницы» или обнуление после чистки — понятная ошибка валидации. 1 конкурент →
`K=1` с пометкой о малой выборке.
