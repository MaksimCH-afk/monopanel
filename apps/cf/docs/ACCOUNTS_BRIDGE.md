# Единая сущность аккаунта Cloudflare («мост»)

Дизайн для дальнейшей работы. Цель — чтобы **мастер-токен**, **выпуск кастомного
токена**, **дашборд**, **аккаунты деплоя** и **домены деплоя** были одной сущностью,
которая живёт и чинится сама — без ручных кнопок «Убрать дубли», «Переклеить
домены», «Обновить имена», «Импортировать».

---

## 1. Как размазана сущность сейчас

Один реальный аккаунт Cloudflare сегодня представлен в 3–4 местах, ничем жёстко
не связанных:

| Таблица | Что хранит | Ключ связи |
|---|---|---|
| `master_tokens` | мастер-токен + рабочий child (`working_token`), `account_email` | нет связи с аккаунтом |
| `cloudflare_credentials` | «аккаунт» = токен/ключ (`api_key`, `auth_type`, `cf_account_uid`) | `id` (может быть **несколько строк на один CF-аккаунт**) |
| `cloudflare_api_tokens` | вручную заведённые scoped-токены | `account_id → cloudflare_credentials.id` |
| `cloudflare_accounts` | домены | `account_id → cloudflare_credentials.id`, `zone_id` |
| `cf_deploy_sites` | сайты деплоя | `account_id → cloudflare_credentials.id` |

**Корень всех недавних багов** — нет канонической сущности «CF-аккаунт», а связи
идут по `cloudflare_credentials.id`, а не по реальному аккаунту Cloudflare:

- Идентичность аккаунта (`cf_account_uid`, имя) резолвилась только через
  `GET /accounts` (право Account Settings:Read). У токенов на 14 прав его нет →
  `uid` пустой, имя = заглушка `token-XXXX`.
- Из-за пустого `uid` дедуп группирует по имени, а не по аккаунту → риск склейки/
  недосклейки.
- `cloudflare_accounts.account_id` фиксируется первой вставкой (`INSERT OR IGNORE`)
  и не переклеивается → домен может «висеть» на кредеталe, чей токен зоной не владеет.
- Деплой смотрит `hasToken` в `cloudflare_api_tokens`, а токен из мастера лежит в
  `cloudflare_credentials.api_key` → «нет токена» при живом токене.

Кнопки-костыли (дедуп, переклейка, обновление имён) — это ручная компенсация
отсутствия единой сущности.

---

## 2. Целевая модель

Канонический аккаунт с ключом = **реальный Cloudflare account id (uid)**. Всё
остальное (токены, домены, сайты) висит на нём.

```
cf_account (id, user_id, cf_uid UNIQUE, name, created_at, updated_at)
   ├── cf_token   (id, cf_account_id, kind: master|child|global, value, perms, status, source)
   ├── cf_domain  (id, cf_account_id, zone_id, name, …)   // бывш. cloudflare_accounts
   └── cf_deploy_sites.cf_account_id
```

- `cf_uid` — единственный ключ идентичности. Один реальный аккаунт = одна строка
  `cf_account`, сколько бы токенов на него ни завели.
- Токены — дети аккаунта (мастер и дочерние в одной таблице, не в трёх местах).
- Домены и сайты ссылаются на `cf_account`, а не на конкретный токен-кредентал.

### Миграция без большого переписывания (рекомендуемый путь)

Не обязательно сразу вводить новые таблицы. Достаточно сделать
**`cf_account_uid` сквозным ключом** над текущей схемой и ввести инвариант:

1. `cf_account_uid` заполнен у КАЖДОГО `cloudflare_credentials` (резолвер ниже).
2. `UNIQUE(user_id, cf_account_uid)` — один кредентал на реальный аккаунт (дубли
   схлопываются автоматически).
3. `cloudflare_accounts.account_id` всегда указывает на кредентал, чей токен
   владеет зоной (переклейка по факту владения).
4. `master_tokens` и `cloudflare_api_tokens` связываются с аккаунтом через
   `cf_account_uid`, а не через отдельные пути.

Новые нормализованные таблицы (`cf_account`/`cf_token`) — идеальный конечный вид,
но его можно оставить на потом; инвариант выше уже убирает все текущие баги.

## 3. Единый резолвер идентичности

Один helper, который вызывается ВЕЗДЕ, где в системе появляется токен:

```php
// Возвращает ['uid'=>..., 'name'=>...] или null. Работает без Account Settings:Read.
function cfResolveAccount($token) {
    // 1) GET /accounts?per_page=1  → result[0].{id,name}
    // 2) fallback GET /zones?per_page=1 → result[0].account.{id,name}
}
```

Это обобщение уже добавленных `mtResolveAccountName()` и логики из
`reissue_domain_token`. Он даёт `uid` + `name` даже для 14-прав токенов.

## 4. Self-healing: точки внедрения (без кнопок)

Идея: **любое** появление или использование токена прогоняется через один upsert,
который поддерживает инвариант. Тогда ручные кнопки становятся не нужны.

```php
// Резолвит аккаунт по токену, upsert cf_account по uid, привязывает токен,
// подтягивает/переклеивает домены по факту владения зоной. Идемпотентно.
function cfUpsertAccountByToken($pdo, $userId, $token, $authType, $kind) { … }
```

Вызвать её в существующих местах (сейчас каждое из них делает свой кусок вручную):

| Поверхность | Функция сейчас | Что добавить |
|---|---|---|
| Сохранение мастер-токена | `add_master` | upsert аккаунта по uid |
| Выпуск child-токена | `create` | upsert + привязка токена к аккаунту |
| «Добавить в панель как аккаунт» | `save_as_account` | uid из резолвера (уже частично есть) |
| «Создать домены» | `create_zones` | домены сразу на канонический аккаунт |
| Импорт доменов | `import_empty` | привязка по владению зоной, а не OR IGNORE |
| Деплой (публикация/зоны) | `deploy_api` | брать токен канонического аккаунта |

Когда все шесть точек ходят через один upsert:
- дубли не появляются (UNIQUE по uid);
- имена всегда реальные (резолвер);
- домены всегда на правильном аккаунте (проверка владения);
- «нет токена» невозможно (токен — часть аккаунта);
- кнопки «Убрать дубли / Переклеить / Обновить имена» превращаются в
  необязательную страховку (можно оставить как ручной ре-скан или убрать).

## 5. Порядок внедрения в 8 шагов (инкрементально, каждый шаг самостоятелен)

1. **Резолвер** `cfResolveAccount($token)` — общий, с fallback на зоны (uid + name
   даже без Account Settings:Read). *(частично есть: `mtResolveAccountName`)*
2. **Бэкфилл** `cf_account_uid` по всем кредеталам через резолвер — идемпотентно,
   батчами, в `dbRetryOnLock`, вне горячих путей (отдельный проход/кнопка, не рендер).
3. **Дедуп по uid** — обобщение текущего `mtDedupComputeMerges`: при непустом uid
   группировать по uid, имя — только тай-брейк. Схлопывает дубли одного CF-аккаунта.
4. **Частичный UNIQUE-индекс** `(user_id, cf_account_uid)` c условием
   `WHERE cf_account_uid IS NOT NULL AND cf_account_uid <> ''`. Создаётся **только
   после шага 3** (иначе упадёт на существующих дублях/пустышках). См. §6.
5. **Авто-переклейка доменов по владению зоной** — логику `relink_domains`
   вызывать автоматически после импорта/выпуска (не только кнопкой).
6. **`cfUpsertAccountByToken()`** и врезка в 6 точек из §4 — лёгкий upsert в короткой
   транзакции (`BEGIN IMMEDIATE` + retry), **только на событиях записи**, не на чтении.
7. **Деплой** читает токен/домены канонического аккаунта *(частично сделано:
   `hasToken`, `cfDeployLoadCredentials`, домены из БД)*.
8. *(опционально)* нормализованные `cf_account` / `cf_token` — **новые** таблицы +
   перенос данных, не in-place `ALTER` (см. §6).

Каждый шаг убирает свой класс багов и не ломает остальное; большой рефактор (шаг 8)
можно отложить.

## 6. SQLite-специфика и миграции

Проект работает на **SQLite** (`sqlite:` PDO), режим **WAL**, `busy_timeout=60000`,
с **фоновыми писателями** (`monitor.php`, `queue_processor.php`, `cron/daily_sync.php`,
`telegram.php`). Отсюда — жёсткие ограничения, которые обязательно учесть при внедрении:

**1. UNIQUE по uid — только частичным индексом и только после дедупа.**
- `ALTER TABLE` в SQLite **не умеет** добавлять UNIQUE к существующей таблице —
  только `CREATE UNIQUE INDEX`.
- Пустая строка ломает уникальность: SQLite считает `'' = ''` (а несколько `NULL`
  разрешены). Поэтому «неизвестный uid» — это **`NULL`, а не `''`** (в коде уже
  используется `COALESCE(NULLIF(?, ''), …)` — так и держать).
- Индекс создаём **после** шага 3 и как частичный:
  ```sql
  CREATE UNIQUE INDEX IF NOT EXISTS idx_cred_account_uid
    ON cloudflare_credentials(user_id, cf_account_uid)
    WHERE cf_account_uid IS NOT NULL AND cf_account_uid <> '';
  ```
  Иначе создание упадёт на существующих дублях/пустышках.

**2. Один писатель → лок-бюджет (главный риск).**
Панель уже борется с «database is locked» из-за фоновых писателей. Self-healing
upsert (§4) добавляет записи в горячие пути, поэтому:
- upsert — **короткий**, в `dbRetryOnLock` + `BEGIN IMMEDIATE` (как в `dedup_accounts`);
- **не запускать** тяжёлый upsert/relink на чисто-читающих путях (рендер, `account_zones`
  GET) — только на явных событиях записи (сохранение мастера, выпуск токена) или
  отдельным throttled-проходом;
- сетевые вызовы (резолв uid) выносить **до** транзакции, внутри держать только запись
  (паттерн уже применён в `dedup_accounts`).

**3. Внешние ключи не enforce-ятся.**
`PRAGMA foreign_keys=ON` в проекте не выставлен → FK в схеме декоративны. Целостность
(аккаунт↔токены↔домены) держим **в коде**, каскады не полагаемся. Включать
`foreign_keys=ON` глобально не стоит — может сломать существующие удаления.

**4. Лимиты ALTER.**
Добавить колонку — можно (проект так делает, идемпотентно в `try/catch`). Сменить
constraint / удалить колонку — нельзя (до SQLite 3.35). Нормализация (`cf_account`/
`cf_token`, шаг 8) — это новые таблицы + копирование, а не in-place `ALTER`.

**5. Миграции — идемпотентно и безопасно под WAL.**
Схемные изменения — через существующий паттерн в `config.php` (`CREATE TABLE IF NOT
EXISTS`, `ALTER … ADD COLUMN` в `try/catch`, `CREATE INDEX IF NOT EXISTS`). Бэкфилл
(шаг 2) — отдельным проходом с сетевыми вызовами вне транзакции и записью в ретрае,
не в момент старта/рендера.

## 7. Что это даёт каждой поверхности

- **Мастер-токен**: сохранил мастер → аккаунт появился с реальным именем и всеми
  доменами, без «Импортировать/Обновить имена».
- **Выпуск кастомного токена**: новый child сразу привязан к тому же аккаунту, не
  плодит дубль-кредентал.
- **Дашборд**: домены всегда под правильным аккаунтом; «Перевыпустить → Авто» не
  нужен как рутина.
- **Деплой (аккаунты)**: список аккаунтов = канонические CF-аккаунты, всегда «с
  токеном».
- **Деплой (домены)**: домены аккаунта берутся из канонической связи (совпадает с
  тем, что реально видит токен), без расхождения «0 vs 1».
