# Модуль «Деплой из ZIP»

Публикация готового **статического сайта из ZIP** на **Cloudflare Workers Static Assets** — без хостинга и сервера. Один сайт = один воркер (имя = домен). Раздел доступен в меню `cf` → **«Деплой»**.

Модуль надстроен над существующим `apps/cf` и переиспускает его аккаунты, токены и хелперы CF API. Существующий код не переписывался.

## Быстрый старт

1. Загрузите ZIP со статикой (корневой `index.html` в корне архива или в единственной верхней папке).
2. Нажмите **«Проверить архив»** — модуль покажет сводку (файлы, размер, страницы, `.htaccess`/`404`, серверные файлы).
3. Выберите **аккаунт Cloudflare** (поиск по email) и **домен**.
4. **«Опубликовать»** — сайт выходит на служебный `*.workers.dev`.
5. При необходимости в блоке **«Мои сайты»** привяжите **Custom Domain + SSL** и настройте версии/мету.

## Токен (важно)

Для деплоя нужен **API-токен с правом `Workers Scripts:Edit`** (и для привязки домена — доступ к зоне/DNS/SSL). Модуль **берёт scoped-токен из раздела «Мастер-токен»** для выбранного аккаунта (таблица `cloudflare_api_tokens`), а не старый Global Key. Если у аккаунта нет токена — создайте его в «Мастер-токен» (кнопка «все права» покрывает нужное). В поле выбора аккаунта есть индикатор наличия токена.

Минимальный набор прав токена: `Account → Workers Scripts:Edit`; `Zone → Zone:Read/Edit, DNS:Edit, SSL and Certificates:Edit, Workers Routes:Edit`.

## Выбор аккаунта и домена

- **Аккаунт** — поле с автодополнением по email. Домен должен принадлежать этому аккаунту, чтобы его можно было привязать.
- **Домен** — после выбора аккаунта подтягиваются его зоны; при вводе показывается, есть ли домен в аккаунте. Ручной ввод домена доступен всегда.

## Приоритет: что если сайт уже задеплоен на другом сервере

Приоритет определяется тем, **куда указывают nameservers/DNS домена**, а не тем, «кто задеплоил».

- **Домен в этом Cloudflare-аккаунте (зона активна, NS → Cloudflare).** Привязка Custom Domain ставит проксируемую (оранжевое облако) запись на воркер. Запрос обслуживается **воркером на edge Cloudflare** и до старого origin-сервера не доходит — воркер имеет приоритет над прежней A-записью/origin.
- **Домен НЕ в этом аккаунте.** Публикация живёт только на `*.workers.dev` и **на боевой домен не влияет** — по нему по-прежнему отвечает текущий сервер. Чтобы переключить, перенесите зону в этот аккаунт (смена nameservers) — тогда действует первый сценарий.
- **Имя воркера** уникально в пределах аккаунта. Повторный деплой того же домена в тот же аккаунт — обновление того же воркера (атомарно, хэш-дедуп неизменённых файлов). В разных аккаунтах воркеры не пересекаются.

UI разрешает привязку домена только когда домен реально в выбранном аккаунте (правило «домен и сайт — в одном аккаунте»).

## Режимы защиты

- **static-only** (по умолчанию) — чистая раздача ассетов; запросы к статике не считаются вызовом воркера и почти не расходуют дневной лимит.
- **worker-first** — воркер срабатывает на каждый запрос (гео/бот-логика на самом сайте). **Каждый заход расходует дневной лимит вызовов аккаунта** (Free — 100 000/день); для большого трафика нужен платный аккаунт.

## Версии и мета (правки по домену)

Модуль хранит исходник корня сайта, поэтому правки не требуют повторной загрузки ZIP:

- **Копия в подпапку** (`/en/`, `/es-cl/`) — самодостаточная (переписывает корне-абсолютные пути) или с общими ассетами корня.
- **canonical / hreflang / x-default** — редактируются по версиям; кластер реципрокный; конфиг хранится в модуле и **переприменяется при каждой публикации** (в т.ч. после повторной загрузки ZIP).

## Лимиты (Cloudflare)

| Лимит | Значение |
|---|---|
| Размер **файла** | 25 MiB (архив в целом — без ограничения) |
| Файлов на сайт | 20 000 (Free) / 100 000 (Paid) |
| Воркеров (= сайтов) на аккаунт | ~100 (Free) / ~500 (Paid) |
| Запросов/день (только worker-first) | 100 000 (Free) / без лимита (Paid) |

## Хранение

- `cf_deploy_sites` — сайты (домен, воркер, аккаунт, режим, статус привязки/SSL).
- `cf_deploy_versions` — версии (корень + подпапки).
- `cf_deploy_meta` — конфиг мета/hreflang.
- Исходники корня сайтов — `/data/cf/deploy_sites/<id>/src` (fallback `apps/cf/deploy_data/`, в `.gitignore`).

## Файлы модуля

- `deploy.php` — страница раздела и UI.
- `deploy_api.php` — AJAX-эндпоинт (валидация, проверка домена, деплой, привязка/отвязка/SSL, версии, мета).
- `deploy_lib.php` — валидация/инвентаризация ZIP.
- `deploy_worker.php` — оркестратор Static Assets и Custom Domains.
- `deploy_edits.php` — хранилище исходника, копии в подпапки, мета/hreflang, сборка.

> ⚠️ Живые вызовы к Cloudflare (upload-session/бакеты/PUT воркера/`*.workers.dev`/привязка домена) написаны строго по официальному протоколу Static Assets и Custom Domains. Первый прогон стоит сделать на тестовом домене — в первую очередь проверить хэш ассетов и получение completion-JWT.
