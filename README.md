# Единая панель (monopanel)

Объединение 6 локальных приложений в **один Docker-контейнер** с дашбордом-плитками.
Каждое приложение работает как есть и слушает свой порт; страница-дашборд (`:333`)
даёт единую точку входа — плитки ведут на порты приложений.

> Приложения **не переписываются**. Объединение — на уровне точки входа (один
> образ + supervisord), а не кода: стеки разнородны (PHP, FastAPI ×2, Flask ×2,
> Next.js ×2), слить их в один код нереально.

---

## Приложения и порты

| Софт | Назначение | Стек | Порт | Маршрут плитки |
|------|-----------|------|------|----------------|
| **дашборд** | страница-плитки | Caddy (статика) | **333** | — |
| **cf** | Cloudflare Panel (домены, SSL, DNS, токены) | PHP 8.1 + Apache | **3331** | `/` |
| **seo** | GSC / SEO Dashboard | Next.js + Flask | **3332** | `/` |
| **ank** | Генератор анкоров (HubNero) | FastAPI + uvicorn | **3333** | `/` |
| **img** | ImageGen (мультимодельная генерация) | Next.js + Prisma | **3334** | `/` |
| **arc** | Web Archive Checker | FastAPI async | **3335** | `/` |
| **skins** | Brandskins (темы брендов) | Flask + gunicorn | **3336** | `/admin` |

- Старых портов (1000/9999/8000/3000/8888) в новой сборке не остаётся.
- Конфликт порта 3000 (seo и img) решён переназначением: seo→3332, img→3334.
- Flask-бэкенд seo (`5001`) — служебный, наружу не публикуется.
- `333` — привилегированный порт (<1024); под Docker обычно ок. Если демон
  откажется привязать — запасной вариант `3330` (поменять в `.env` и `docker-compose.yml`).

### Исходники приложений (вендоринг)

Код всех 6 приложений **втянут прямо в этот репозиторий** (папки `apps/*`), а не
подключён сабмодулями — чтобы всё жило в одном сборном репозитории и не нужно было
ходить по отдельным репам. В каждой папке лежит родной `README.md` приложения.

| Папка | Источник | Стек | Запуск в образе |
|-------|----------|------|-----------------|
| `apps/cf`    | `MaksimCH-afk/CF-panel`         | PHP 8.1 + Apache | Apache :3331 + фоновые queue/monitor |
| `apps/seo`   | архив `SEO-Dashboard`           | Next.js + Flask  | `next start` :3332 + `backend_api.py` :5001 |
| `apps/ank`   | `MaksimCH-afk/generator-ankorov`| FastAPI          | `uvicorn app.main:app` :3333 |
| `apps/img`   | `MaksimCH-afk/pictures`         | Next + Prisma    | `prisma db push` + `next start` :3334 |
| `apps/arc`   | `MaksimCH-afk/webarhive`        | FastAPI          | `alembic upgrade` + `uvicorn …factory` :3335 |
| `apps/skins` | `MaksimCH-afk/brandskins1`      | Flask + gunicorn | `gunicorn app:app` :3336 (`/admin`) |

Что поменяли при интеграции (минимально, логику не трогали):
- **порты** переведены на раскладку 3331–3336 (Apache/uvicorn/next/gunicorn);
- **cf**: фоновые обработчики (`queue_processor.php`, `monitor.php`) перенесены из
  родного `docker-entrypoint.sh` в `supervisord` и нацелены на `:3331`;
- **seo**: бэкенд `backend_api.py` теперь слушает `0.0.0.0` (одна строка) — иначе
  публикация порта из Docker не работает;
- **данные** вынесены в тома `/data/*` (cf через симлинк, ank/arc/img через env).

### Как образ устроен внутри

- База `php:8.1-apache`; поверх ставятся Node 22, Python 3, tesseract, Caddy.
- У каждого Python-приложения свой venv (`/venv/ank`, `/venv/arc`, `/venv/seo`,
  `/venv/skins`) — из-за конфликта версий Flask (seo 2.3 / skins 3.1).
- `supervisord` держит все процессы, перезапускает упавшие, логи по каждому
  отдельно (префикс `[program:*]`).
- Дашборд — статика на Caddy (`:333`); плитки берут хост динамически (работает
  и по `localhost`, и по IP).
- **Дизайн дашборда** перенесён из приложения **webarhive** (Web Archive Checker):
  единая дизайн-система (дизайн-токены, тёмный топбар с лого, карточки-плитки,
  шрифты Inter / JetBrains Mono) и **переключатель тем** (стандарт / лён / nord /
  сланец / океан) с сохранением выбора в `localStorage`.

## Запуск

```bash
cp .env.example .env       # по желанию — ключи API (OpenRouter/OpenAI/Auth0)
docker compose up --build  # первая сборка долгая: 3 рантайма + 2 Next-сборки
```

Открыть дашборд: **http://localhost:333**

Логи: `docker logs -f panel` (помечены `[program:...]`).

## Перенос данных (только cf и ank)

Сохраняются данные **cf** (домены, аккаунты, токены, группы) и **ank** (проекты,
стратегии, профили). Остальные приложения стартуют с чистого листа.
Подробный порядок забора и размещения БД — в [`data/README.md`](data/README.md).

> ⚠️ `cloudflare_panel.db` содержит API-токены Cloudflare. Папка `data/` —
> в `.gitignore`, в репозиторий не коммитится.

## Безопасность

- Секреты не коммитятся: `data/`, `*.db`, `credentials.txt`, `.env` закрыты
  в `.gitignore`.
- Если GitHub PAT когда-либо попадал в чат/историю — считать его
  скомпрометированным и **отозвать** (GitHub → Settings → Developer settings →
  Personal access tokens → Revoke), даже если scope ограничен одной репой.
- Общий логин не включён — каждое приложение открыто на своём порту. При
  необходимости добавляется слой Caddy с одним паролем перед портами.

## Честные оговорки

- **Первая сборка тяжёлая и долгая**, образ крупный — осознанный компромисс ради
  «одного контейнера» для single-user инструмента.
- Это **первый рабочий каркас**, не вылизанный билд. Большой мульти-рантайм образ
  обычно требует 1–2 точечных правок под конкретную машину (версии пакетов,
  npm-зависимости). Падение на шаге сборки — ожидаемо, чинится в
  `Dockerfile` / `docker/supervisord.conf`.
- **seo + Auth0**: для входа нужны домен/ключи в `.env`; без них фронт поднимется,
  но вход не сработает. Сборка seo обёрнута в `|| WARN` — если `next build` упадёт
  из-за окружения, образ всё равно соберётся, а seo можно будет починить отдельно.
- **seo-бэкенд на `:5001` публикуется наружу.** Фронт seo обращается к бэкенду по
  жёстко зашитому `http://localhost:5001` **из браузера**, поэтому порт 5001 в
  `docker-compose.yml` проброшен на хост (первоначальный план «5001 только внутри»
  не сработал бы). GSC-вход требует настройки Google client_secret (см. apps/seo).
- **cf**: на чистой БД создаётся админ **admin / admin** (`docker-seed.php`) —
  смените пароль после первого входа. БД и `credentials.txt` лежат в томе `/data/cf`.
- **skins** хранит состояние (активный шаблон, `out/`) в файлах внутри образа
  (`SITE_ROOT=/srv/panel/apps/skins`) — между пересборками сбрасывается; при
  необходимости примонтируйте `apps/skins` томом.
- **arc** стартует с чистого листа (миграции alembic на первом запуске); БД в `/data/arc`.
- **Дашборд — гибрид iframe + новая вкладка.** Плитка открывает приложение
  iframe-вкладкой прямо в дашборде; на каждой плитке есть кнопка ↗ «в отдельной
  вкладке». iframe-вкладки переживают переключение (создаются лениво и хранят
  состояние).
- **iframe-нюансы по приложениям:**
  - `ank`, `arc`, `skins` — встраиваются как есть.
  - `cf` — встраивается после снятия `X-Frame-Options`/CSP на стороне его Apache
    (`docker/cf-frame.conf`).
  - `img` — Next.js по умолчанию не блокирует фрейминг; встраивается как есть.
  - `seo` — помечен **«только новая вкладка»**: вход через сторонний домен Auth0
    и сторонние куки внутри iframe не работают, поэтому seo всегда открывается
    отдельной вкладкой (плитка делает это автоматически).

## Следующие шаги

- Пройти первую сборку на целевой машине и поправить, если что-то отвалится
  (версии пакетов/npm — ожидаемо для мульти-рантайм образа).
- При желании — общий логин (один пароль на все плитки) через слой Caddy.
- При желании — попытаться встроить и `seo` (например, отдельный сервис-аккаунт
  вместо Auth0, или прокси-обёртка с подменой куки) — сейчас он только в новой вкладке.
